## Metadata

- Full Title: The Case Against Documentation
- Category: #articles
- Summary: Documentation isn't always beneficial, as it takes time to maintain and can become outdated. It's important to consider the return on investment and focus on documentation that doesn't change often, is widely needed, or is already written. Inline documentation with the code is preferred, and future advancements may allow for more intuitive access to information.

## Highlights

- Here are some cases where the benefits can outweigh the costs: ([View Highlight](https://read.readwise.io/read/01ja0ehzpfxak16cjnyaj1j3bh))
- **Documentation that doesn’t change often** - We get more value from this since it’ll get more readership over its lifetime and costs less to maintain. For example, high-level system diagrams are often worthwhile. ([View Highlight](https://read.readwise.io/read/01ja0ej4qjwvjtmvtd175fk0qx))
- **Documentation that many people will need** - The impact of documentation scales with the number of people who read it. Documenting a popular API or library is well worth the time if thousands of engineers need it. ([View Highlight](https://read.readwise.io/read/01ja0ej8bn5q58tq3y1eqjrvcm))
- **Documentation you already have** - I’m a big fan of [writing for clearer thinking](https://www.developing.dev/p/why-engineers-need-to-write). If you already write for yourself, publishing it for others can be worth it since it’s near zero cost with some benefit. ([View Highlight](https://read.readwise.io/read/01ja0ejf1vkeb09xy4gxkjd8fw))
- my favorite documentation is inline with the code. It’s much easier to keep up to date and find when needed when it’s next to the code it describes. ([View Highlight](https://read.readwise.io/read/01ja0emph0b6a7dj636xqnf2db))
